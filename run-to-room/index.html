3개 게임을 잇는 데이터 흐름

외부 게임 → 건물 번호 수신 → 맵 컨셉 분기 → 매니저들이 각자 역할 수행 → 결과 데이터를 다음 게임으로 송신. 시그널/postMessage 채널을 따라 데이터가 한 방향으로 흐르도록 설계했습니다.

// cross-game data flow · signal pipeline

// 1. 외부 게임 → RunToRoom (입장 시 건물 번호 수신)
[External Game A]
  → Redbrick Signal { buildingId: 3 }
  → postMessage / LocalStorage
        ↓
[RunToRoom Boot]
  → SignalReceiver.on('buildingData')
  → MapGenerator.build(buildingId)
        ↓
// 2. 내부 시스템 — 매니저 분리 구조
  MapGenerator
    → ItemManager       // 엑셀 확률표 기반 스폰
    → EnemyManager      // 건물별 적 패턴
    → EventManager      // 중간 이벤트 트리거
        ↓
// 3. 게임 종료 → 다음 게임으로 결과 데이터 송신
[RunToRoom End]
  → ResultCollector { items: [...], buildingId }
  → Redbrick Signal emit('survivorResult')
  → LocalStorage persist
        ↓
[External Game B] — 받아서 자기 게임에 반영
MY WORK · 직접 설계한 4가지 핵심 시스템

자동 맵 · 매니저 분리 · 엑셀 밸런싱 · 외부 데이터

합작 프로젝트 중 본인이 맡은 캐주얼 생존 게임 파트에서 직접 설계·구현한 핵심 영역입니다. 각 영역은 서로 독립적으로 동작하지만, MapGenerator를 중심으로 하나의 흐름으로 묶여 있습니다.

AREA 01 · MAP GENERATOR

자동 맵 생성기

건물 번호를 입력으로 받아 맵 컨셉·블록 배치·장애물 배치를 절차적으로 구성. 동일 prefab을 회전·재배치하여 6개 컨셉을 표현.

  • 입장 시마다 다른 레이아웃 — 반복성 제거
  • buildingId → 컨셉 매핑 테이블
  • 장애물 주변 예외 처리 — 막힌 구조·고립 영역 방지
  • 블록 prefab 재활용으로 6컨셉을 1세트 자산으로 표현
AREA 02 · MANAGER SEPARATION

매니저 분리 구조

아이템·적·중간 이벤트를 각각 독립 매니저로 분리. 테스트 및 룰 변경 시 한 모듈만 수정하면 되도록 책임을 명확히 가름.

  • ItemManager — 아이템 스폰 / 회수 / 인벤토리
  • EnemyManager — 적 스폰 · 추격 AI · 난이도 곡선
  • EventManager — 중간 이벤트 트리거 · 보너스 페이즈
  • 각 매니저는 MapGenerator로부터 컨셉 정보만 받음
AREA 03 · EXCEL-DRIVEN BALANCE

엑셀 기반 밸런싱

아이템 확률·등장 횟수를 엑셀 시트로 관리하고, 게임 빌드 시 JSON으로 변환해 로딩. 코드 수정 없이 밸런스 패치가 가능한 파이프라인.

  • 엑셀 시트 → JSON 변환 (빌드 파이프라인)
  • 아이템별 probability, maxCount, buildingFilter
  • 기획자/디자이너가 직접 시트만 수정해도 반영
  • 건물별 등장 풀(pool) 분리로 컨셉 일관성 확보
AREA 04 · CROSS-GAME DATA

게임 간 데이터 수신·송신

Redbrick Signal과 postMessage API를 함께 사용해 다른 웹 게임으로부터 건물 번호를 받고, 플레이 결과를 다음 게임으로 전달.

  • SignalReceiver — 외부 게임의 건물 데이터 수신
  • ResultCollector — 획득 아이템 누적 · 직렬화
  • LocalStorage 폴백 — 시그널 미수신 시 백업 채널
  • 건물 데이터 부재 시 랜덤 컨셉으로 안전 폴백

매니저 분리 구조 — 의사 코드

실제 코드는 GitHub에 공개되어 있지만, 여기서는 설계 의도를 한눈에 보여주기 위해 핵심 인터페이스만 간추렸습니다.

map_generator.js · pseudo JS 
// 건물 번호 → 맵 컨셉 매핑 + 매니저 초기화
class MapGenerator {
  build(buildingId) {
    const concept = this.resolveConcept(buildingId);  // 1~6

    this.placeBlocks(concept.layout);         // prefab 재활용
    this.placeObstacles(concept.obstacleRule); // 주변 예외 처리

    // 매니저는 컨셉 정보만 받아 각자 초기화 — 결합도 최소
    ItemManager.init(concept.itemPool);     // 엑셀 → JSON
    EnemyManager.init(concept.enemyPattern);
    EventManager.init(concept.eventSchedule);
  }
}
cross_game_bridge.js · pseudo JS 
// 외부 게임 → 본 게임 (입장 시) / 본 게임 → 다음 게임 (종료 시)
SignalReceiver.on('buildingData', (data) => {
  const id = data?.buildingId ?? LocalStorage.get('buildingId');
  MapGenerator.build(id ?? Random.pick(1, 6)); // 안전 폴백
});

function onGameEnd(result) {
  const payload = {
    items: ResultCollector.serialize(),
    buildingId: Game.currentBuildingId,
  };
  Signal.emit('survivorResult', payload);
  LocalStorage.set('survivorResult', payload); // 백업
}

실제로 마주친 문제와 해결

합작 프로젝트, 처음 시도해보는 게임 간 통신, 그리고 인턴 기간이라는 시간 제약 — 각각의 문제를 어떻게 풀었는지 정리했습니다.

Repetitive Map
고정된 맵의 반복성 — 한두 번 플레이하면 금방 질림
solution
입장 시마다 절차적으로 맵을 생성. 장애물 배치 시 주변에 또 다른 장애물이 생성되지 않도록 예외 처리해 좁은 통로·고립 영역을 방지하고 플레이 친화성 확보.
6 Concepts × Cost
6개 건물 컨셉을 각자 만들면 자산·작업량이 폭증
solution
블록 prefab을 재활용하고, 회전·재배치·머티리얼 스왑으로 컨셉만 갈아끼우는 방식 채택. 자산 1세트로 6개 분위기를 표현해 인턴 기간 내 완성도 확보.
Balance Patch
밸런스 패치를 위해 매번 코드를 수정해야 하는 비효율
solution
엑셀 확률 테이블 도입 — 빌드 파이프라인에서 JSON으로 변환해 로딩. 기획자·디자이너가 시트만 수정해도 반영되므로 개발자 병목 제거.
First-time Cross-Game
다른 웹 게임과의 데이터 연결이 처음 — 안정성 보장이 어려움
solution
Redbrick Signal을 1차 채널로, postMessage + LocalStorage를 폴백으로 구성. 시그널이 도달하지 않거나 늦더라도 게임이 멈추지 않고 랜덤 컨셉으로 안전하게 시작.
Internship Time Box
프로젝트 규모가 인턴 기간 내 완성하기에는 과도
solution
완성보다 핵심 시스템 (자동 맵 · 매니저 분리 · 데이터 연동)의 구조를 먼저 갖추는 데 집중. 추후 누가 이어받아도 확장할 수 있도록 인수인계 문서 + GitHub 공개.
