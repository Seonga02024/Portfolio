// architecture

5-layer separation, top-down dependency

UI는 입력과 렌더링만, Engine은 순수 게임 규칙만. 그 사이를 Protocol 패킷이 일관되게 가로지릅니다. 싱글플레이와 멀티플레이가 동일한 코드 경로를 공유합니다.

LAYER 01 · Top

UI Layer

Scripts/ui · Scripts/client

비주얼 렌더링, 입력 처리, DOTween 애니메이션.

LAYER 02

Network Layer

Scripts/network

Protocol 라우팅, CPacket 바이너리 직렬화, RPC 브리징.

LAYER 03

Room Layer

Scripts/CGameRoom.cs

서버측 오케스트레이션, 턴 소유권 검증, 프로토콜 디스패치.

LAYER 04

Engine Layer

Scripts/engine

순수 게임 규칙. 카드 분배, 점수 계산(stateless). Unity 의존성 0.

LAYER 05 · Bottom

AI Layer

Scripts/ai

Evaluation + MCTS. ScriptableObject 기반 가중치 튜닝.

MY CONTRIBUTIONS · 내가 작업한 영역

직접 설계 · 구현한 4가지 핵심 영역

UGS 멀티플레이 안정화부터 로그인 플랫폼 통합, 세션-재화 동기화까지. 프로덕션 환경에서 마주친 실제 문제들을 해결하며 게임의 안정성을 끌어올린 작업들입니다.

AREA 01 · MULTIPLAYER STABILITY

UGS 멀티플레이 안정화

Distributed Authority 세션 기반 1:1 매칭에서 발생하는 호스트 마이그레이션, 연결 끊김, 세션 검증 등 엣지 케이스를 안정화.

  • 호스트 마이그레이션 시 disconnect 상태 전이 및 승자 판정 안정화
  • UpdatePeerHeartbeat 도입으로 무응답 클라이언트 조기 감지
  • 들어올 때 세션 검증 → 실패 시 AI 매칭으로 자연 폴백
  • disconnect 송신 위치 일원화 (호스트/비호스트 양쪽 모두)
AREA 02 · LOGIN PLATFORM

로그인 / 플랫폼 통합

케미·하이보미·게스트 등 다중 플랫폼 로그인 흐름을 통합하고, 플랫폼별 닉네임·랭킹·재화 정책의 일관성을 확보.

  • LoginTokenReader 패키지 오류 및 errorCode 매핑 정비
  • 케미 로그인 시 닉네임 표기 일원화 (사용자 입력 우선)
  • 게스트 모드 닉네임 고정 + 랜덤 닉네임 폴백 로직
  • 랭킹보드 권한 분리 — 케미 사용자만 노출
AREA 03 · SESSION & ECONOMY

세션-재화-게임 흐름 통합

방 입장부터 게임 시작까지의 검증 시퀀스를 재설계하여, 재화 동기화 누락과 비정상 시작을 차단.

  • Spawn 대기 → 프로필 교환 → 서버 재화 검증 → 게임 시작 파이프라인
  • 충전 이후 방 상태 갱신 동기화
  • 게임 종료 후 재화 미반영 및 프로필 시점 불일치 이슈 해결
  • 첫 뻑 보너스 재화 동기화 개선
AREA 04 · RESOURCE & UI INTEGRATION

2차 리소스 통합 + UI 이슈 정비

디자인팀에서 전달된 2차 제작 리소스를 게임 전반에 통합하고, 그에 맞춰 발생한 UI 정합성 이슈를 수정.

  • 2차 제작 리소스 적용 및 설정 변경
  • 프로필 타이틀 · 프로필 0로보 설정 등 메타 데이터 정비
  • FailErrorCodeText 등 시니어 UX 친화 문구 다듬기
  • 매칭 시간 조정 등 UX 파라미터 튜닝
AREA 05 · UGS CLOUD SERVICES

UGS Cloud 서비스 통합

Unity Gaming Services 의 4개 모듈을 게임 전반에 통합하여 크로스-디바이스 데이터 동기화, 서버측 검증, 라이브 운영 채널을 구축.

  • UGS Cloud Save — 점수·재화·프로필 크로스-디바이스 동기화
  • Cloud Code — 재화 증감/출석 체크 서버측 검증 (클라이언트 변조 차단)
  • UGS LeaderBoard — 케미 사용자 전용 실시간 랭킹 보드
  • Remote Config — 인앱 업데이트 + 게임 수치 라이브 조정 (재배포 불필요)

// 싱글톤 Awake 순서 의존성 제거 — 외부 진입점에서 lazy init 보장
public class CUIManager : CSingletonMonobehaviour<CUIManager>
{
    readonly Dictionary<UI_PAGE, GameObject> ui_objects = new();
    bool initialized;

    private void Awake() => EnsureInitialized();

    void EnsureInitialized()
    {
        if (this.initialized) return;
        this.initialized = true;

        TryRegister(UI_PAGE.MAIN_MENU,    "main_menu");
        TryRegister(UI_PAGE.POPUP_GO_STOP, "popup_gostop");
        // ... 30개의 UI 페이지를 단일 진입점에서 등록
    }

    public GameObject get_uipage(UI_PAGE page)
    {
        EnsureRegistered(page);  // 호출 순서 무관 — Awake보다 먼저 와도 안전
        return ui_objects.TryGetValue(page, out var obj) ? obj : null;
    }
}

Assets/Scripts/network/MultiplayerBridge.cs C# 
// 호스트/클라이언트 양쪽이 동일한 로컬 서버 인터페이스로 보이도록 추상화
public class MultiplayerBridge : NetworkBehaviour
{
    [Rpc(SendTo.Owner)]
    public void SubmitInputRpc(PROTOCOL protocol, byte[] payload)
    {
        if (!IsSessionOwner) return;
        CLocalServer.Instance.OnReceiveFromRemote(protocol, payload);
    }

    // 매칭 실패 시 AI 폴백 — 3초 대기 후 로컬 AI로 자연 전환
    async UniTask FallbackToAiAfterTimeout()
    {
        await UniTask.Delay(3000);
        if (!IsMatched) CLocalServer.Instance.SpawnAiOpponent();
    }
}
// problem_solving

Real problems, shipped solutions

본인 작업 영역에서 실제로 마주친 문제와 해결 과정. git 히스토리에서 발췌한 사례입니다.

Host Migration
매칭 및 호스트 마이그레이션 시 승자 판정이 어긋나 결과 화면이 깨짐
solution
마이그레이션 시점에 disconnect 상태 전이를 명확히 분리하고 승자 판정 로직을 안정화. 호스트/비호스트 양쪽이 동일하게 disconnect를 송신하도록 위치 일원화.
Session Validation
방 입장 직후 검증 실패로 비정상 진입 시 재화·매칭이 어긋남
solution
입장 시 세션 검증 단계를 도입. 실패하면 재화를 갱신한 뒤 AI 매칭으로 자연 폴백. 시작/종료 위치를 재정렬해 안정화.
Game Start Pipeline
Spawn 직후 게임이 시작되어 재화/프로필 정보가 동기화되지 않음
solution
Spawn 대기 → 프로필 교환 → 서버 재화 갱신 → 게임 시작 검증의 4단계 파이프라인을 명확히 분리. 각 단계의 실패는 재시도 또는 폴백으로 분기.
Init Order
CUIManager 싱글톤 Awake 순서 의존으로 NullReferenceException
solution
필드 선언 시점 초기화 + lazy init 패턴. 외부 호출 진입점에서 EnsureInitialized 호출로 호출 순서와 무관하게 안전.
